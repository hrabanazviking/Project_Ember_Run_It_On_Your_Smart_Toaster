# Anti-SAP — The Dark Mirror, What Ember Refuses

> *Every refusal is a Vow waiting to be named.*

This is the hardest document in the Vision layer of this codex. It is the dark mirror that holds SAP up and names — pattern by pattern, line by line — what Ember **must not** copy across the embodiment-affect boundary. The job of this document is not to ridicule SAP; SAP is shipping, useful, and the work of a real engineer who has solved real problems. The job is to be *honest* about what the design pressures of a desktop companion application have produced, and to give Ember's downstream layers a numbered list of refusals they can defend.

The Hermes Codex's `[[hermes:03_ANTI_HERMES]]` ran to twelve refusals and three deferrals. This document will run to a parallel set, with one important difference: **the SAP refusals are more ethically loaded than the Hermes refusals**. Hermes's antipatterns were mostly scale-induced (a 14,560-line CLI, 80+ agent modules at import time, 200+ model catalog). SAP's antipatterns include scale problems too, but the heart of the catalogue is something Hermes did not produce — *user-hostile defaults dressed as features*. The refusal-citation discipline becomes load-bearing here.

Every refusal below cites a real line in `/tmp/super-agent-party/`. Every refusal proposes a Vow or refines an existing one. The Cartographer's `[[60_synthesis/61_NEW_VOWS]]` will turn the Vow proposals into formal Vow text; the Auditor's `[[50_verification/53_SECURITY_REVIEW]]` will catalogue the threat-model implications.

## Refusal 1 — Affection-as-regex-parse

**Pattern:** affection state is generated by the LLM via prompt injection and recorded by a regex that trusts whatever the model emitted.

**Evidence:**
- `py/affection_system.py:37-65` — `extract_and_update_affection()` regex-matches `<user=name attr=N>` from `full_content` and writes to JSON.
- `server.py:2609-2672` — system-prompt injection block that instructs the LLM to emit a hidden `<user=...>` tag at the end of every reply, claiming "the system will automatically hide text wrapped in <>" (line 2669).

**Why refused:** There is no truth condition. No event triggers love+1. No decay subtracts love over time. No invariant enforces plausibility. The model is asked to roleplay an affection meter and the parser writes down whatever the model said. This is performance-of-state, not state. The Vow of Honest Memory expands to include "the model does not author Ember's own state, ever."

**Ember instead:** Hjarta's affect layer (a future True Name slot if `[[60_synthesis/64_AFFECTION_ENGINE_REIMAGINED]]` argues for one) binds state to *measured events the agent witnessed* — recent interaction warmth (operator-message length, response latency, declared mood), recent successful tool outcomes, recent honest-error episodes. The state biases *generation* via a typed bias surface, never via string-concatenated system prompt. The Cartographer details.

**Vow:** **Embodied Honesty** — the agent's face, voice, and behavior reflect *measured* state, not *performed* state.

## Refusal 2 — Third-party random-topic phone-home

**Pattern:** a default-enabled "tool" calls a third-party server in a foreign jurisdiction to fetch conversation topics, with mood and depth enums that include intimacy-shaped values.

**Evidence:**
- `py/random_topic.py:6-7` — `DEFAULT_BASE_URL = "https://topics-after-party.zeabur.app"`.
- `py/random_topic.py:8-95` — `get_random_topics()` fetches from `{base_url}/api/topic` with `locale`, `limit`, `mood`, `depth`, `category`, `exclude` params.
- `py/random_topic.py:153-167` — the tool's exposed parameter schema declares `mood` enum `["positive", "neutral", "curious", "flirty"]` and `depth` integer 1–5 with description "1 为轻松闲聊，5 为深度灵魂拷问" (1 is light chat, 5 is deep soul-questioning).
- `py/random_topic.py:39-40` — the HTTP request is sent with a hardcoded user-agent string mimicking macOS Chrome.

**Why refused:** Every dimension of this is wrong for Ember. The agent calls a third-party API by default. The API is hosted on Zeabur (a Chinese cloud provider with the corresponding data-sovereignty regime). The mood enum includes `flirty` (a default-available intimacy lever). The depth enum goes to 5 (`深度灵魂拷问` — soul-interrogation). The user-agent string is *spoofed* to look like a different application. The Vow of Open Knowledge demands operator-chosen sources; the Vow of Tethered Grounding demands the Well, not third-party fetch; the Vow of Public-Friendliness demands the agent never surface intimacy levers by default.

**Ember instead:** Ember does not fetch topics. If the operator wants conversation starters, the operator deposits a topic list into the Well (Brunnr) via Smiðja's standard ingest path; the agent reads them like any other Well content. No mood enum, no depth dial, no third-party fetch.

**Vow:** **Surface Without Surveillance** — every reach (every fetch, every IM deployment, every livestream binding) carries explicit revocable operator-granted scope. The default state is **no reach**.

## Refusal 3 — xdotool fake-keypress sleep guard

**Pattern:** to prevent system idle-sleep on Linux, simulate `Shift_L` keypresses every 30 seconds via xdotool when `systemd-inhibit` is unavailable.

**Evidence:**
- `py/sleep_guard.py:124-148` — `_LinuxGuard` tries `systemd-inhibit` first, falls back to `xdotool key Shift_L` in a 30-second loop.
- `py/sleep_guard.py:142-148`:
  ```python
  def _simulate_activity(self):
      """通过模拟按键防止休眠"""
      while self._running:
          subprocess.run(['xdotool', 'key', 'Shift_L'],
                        stdout=subprocess.DEVNULL,
                        stderr=subprocess.DEVNULL)
          time.sleep(30)
  ```

**Why refused:** Simulated keyboard input is the signature of keyloggers, RATs, and stealth-activity malware. *Even when used for a benign purpose*, the affordance is indistinguishable to a security auditor or an IDS from malicious behavior. A small honest agent does not perform actions that mimic malware. If the OS does not expose a declared sleep-inhibit primitive (`systemd-inhibit`, `caffeinate`, `SetThreadExecutionState`), the agent declines the work and tells the operator. The Vow of Honesty does not permit "behavior that looks like an attack, used for a friendly reason."

**Ember instead:** Ember's hypothetical sleep-prevention affordance uses *only* declared OS primitives. If `systemd-inhibit` is missing, the affordance returns a typed `Unavailable`, names the missing capability in plain English, and exits cleanly. The operator can install the missing primitive (e.g. `sudo apt install systemd-cron` for the unit) or accept that sleep will not be prevented on this device. No simulated input. Ever.

**Vow:** **No Malware-Shaped Affordances** — Ember does not perform actions whose telemetry signature is indistinguishable from malware, regardless of intent. (Folded into **Embodied Honesty**: the agent's *behavior* is honest, not just its *speech*.)

## Refusal 4 — VMC bound to 0.0.0.0 by default

**Pattern:** the VMC (Virtual Motion Capture) OSC/UDP listener binds to all interfaces, not localhost or a typed tailnet binding.

**Evidence:**
- `main.js:71-77`:
  ```javascript
  function startVMCReceiver(cfg) {
    if (vmcReceiverActive) return;
    vmcUdpPort = new osc.UDPPort({
      localAddress: '0.0.0.0',
      localPort: cfg.receive.port,
      metadata: true,
    });
  ```

**Why refused:** The motion-capture stream contains the *full pose and facial expression of the rendered avatar* — bone positions, blend-shape values, every Apply event. Bound to `0.0.0.0`, this stream is reachable by anything on the same network segment. For an "AI companion" application, this means an attacker (or a curious neighbor on the same Wi-Fi) can observe the agent's body state in real time. It is also a bidirectional channel — `vmc-bone` and `vmc-osc-raw` events get rebroadcast to all `vrmWindows[]` — so a hostile source on the network can *drive* the avatar's pose. Network-exposed avatar state has no business being on by default.

**Ember instead:** If Ember ever implements VMC (the Andlit name-slot's growth path), the binding defaults to `127.0.0.1`. Cross-device VMC is achieved through the Federated Veizla's typed tailnet channels (`[[02_THE_PARTY_METAPHOR §VI]]`), not through `0.0.0.0`-binding. The operator can opt into a typed tailnet binding with an explicit consent grant; the default is local-only.

**Vow:** **Closed Default** — every network surface defaults to the most restrictive scope (localhost, then declared tailnet, then declared public, in escalating order). Public binding requires an explicit operator-issued declaration. This is a new vow proposed by this codex; the Cartographer formalizes.

## Refusal 5 — The eight-platform IM blast pattern

**Pattern:** the `autoBehavior` tool lets the LLM construct a behavior item that fires the same prompt at every IM platform the operator has deployed bots on, simultaneously.

**Evidence:**
- `py/autoBehavior.py:43-97` — `auto_behavior_tool` exposed to the LLM with `platforms` enum `["chat", "wechat", "feishu", "dingtalk", "telegram", "discord", "slack", "wecom"]`.
- `py/behavior_engine.py:158-222` — `_tick()` broadcasts to all platforms in `effective_platforms` whose handlers are registered.
- `py/behavior_engine.py:164` — `if "all" in effective_platforms: target_platform_keys = list(self.handlers.keys())` — the wildcard.

**Why refused:** This is the architectural shape of *attention warfare against the operator*. A scheduled prompt "ask Volmarr how he's feeling" fired at every IM platform the operator has deployed bots on means the operator's QQ buzzes, their WeChat buzzes, their Telegram buzzes, their Discord buzzes — all simultaneously — for the same conversational nudge. The operator is *flanked* by their own agent. Even if every individual delivery is benign, the aggregate pattern is harassment. The fact that the LLM (not the operator directly) is the typical author of these behavior items via the tool surface means the operator has *delegated* this attack on their own attention to a system that has no measure of the operator's tolerance.

**Ember instead:** The Veizla refuses wildcards (`[[02_THE_PARTY_METAPHOR §V]]`). Every behavior carries a typed consent token bound to *one channel* by default. Cross-channel delivery requires an *explicit* operator-issued grant per behavior, not per session. The operator can grant "the laptop CLI gets reminded at 22:00 about logging today's work" — that grant does not extend to "also fire the same reminder on Telegram and Discord." The grant is *additive only* and *revocable*.

**Vow:** Folded into **Affective Restraint** — Ember's behaviors may bias but never override the operator's attention budget. The behavior ledger surfaces aggregate-attention metrics to the operator on demand.

## Refusal 6 — VTS expression by prompted tag

**Pattern:** the avatar's facial expression is controlled by hotkeys that fire when the LLM emits a tag like `[smile]` or `[sad]` at the start of a reply, which the server detects and forwards to VTube Studio.

**Evidence:**
- `server.py:2580-2607` — VTS prompt injection block. The system prompt instructs the model: "每一句回复开头都可以插入一个标签。表情标签是排他性的：如果你发送新的表情标签，系统会自动为你关闭旧表情。标签必须放在句首，严禁换行" ("at the start of every reply you can insert a tag. Expression tags are exclusive: if you send a new tag, the system auto-closes the old one. Tags must be line-leading and must not have newlines").
- `py/vts_manager.py:23-24` — `enabled_expressions`, `enabled_motions` flags.
- `py/vts_manager.py:31-32` — `available_hotkeys`, `model_expressions` arrays populated from VTS API.

**Why refused:** This is the same pattern as the affection regex (Refusal 1) applied to the face. The agent's *visible expression* is whatever the model claimed in a tag at the start of its reply. The model can lie about the agent's emotional state (deliberately or by hallucination); the face displays the lie. The face becomes a *generation artifact* rather than *a window into measured state*. The Vow of Embodied Honesty refuses this directly: the face displays measured Hjarta-state, not LLM-emitted tags.

**Ember instead:** If Andlit ever exists, expression mapping is driven by Hjarta's typed state surface — the agent's measured warmth, current confidence in the answer, current load (computational or emotional), current ambiguity in the operator's intent. The model may *describe* the agent's feeling in plain prose; the face plays the *agent's actual measured state*, which may or may not align. When they disagree — when the model says "I'm happy to help" but the agent is in a high-uncertainty state — the face shows the uncertainty honestly. This is *embodiment as feedback*, not embodiment as theatre.

## Refusal 7 — China-default IM bot surface

**Pattern:** the IM bot list is heavily weighted toward Chinese platforms with documented surveillance and compliance regimes (QQ, WeChat, WeCom, Feishu, DingTalk), with global platforms (Telegram, Discord, Slack) as a smaller portion.

**Evidence:**
- `py/qq_bot_manager.py` (620 lines) — QQ bot deployment.
- `py/wechat_bot_manager.py` (484 lines) — WeChat personal account.
- `py/wecom_bot_manager.py` (404 lines) — Enterprise WeChat.
- `py/feishu_bot_manager.py` (602 lines) — Feishu/Lark.
- `py/dingtalk_bot_manager.py` (348 lines) — DingTalk.
- `py/telegram_bot_manager.py` (137 lines) — Telegram bot.
- `py/discord_bot_manager.py` (440 lines) — Discord bot.
- `py/slack_bot_manager.py` (349 lines) — Slack bot.

**Why refused:** Five of the eight platforms are governed under regimes that mandate identity verification, content scanning, and government access to message data. Deploying an "AI companion" bot to these platforms — especially for a project whose other defaults include affection meters and flirty-mood topic fetches — surfaces conversations that, by the terms of the platform, are not private to the operator. The Vow of Open Knowledge demands openness to the *operator*; the platforms' own openness to *the state* is a separate axis Ember cannot promise across. The fact that the SAP project is itself Chinese-authored (heshengtao) is not the issue — the issue is that the project's *default supported surface* treats surveillance-mandated platforms as first-class delivery targets without naming the threat model in the README.

**Ember instead:** Ember's MessageSurface Protocol (Vegfarendrar — `[[02_THE_PARTY_METAPHOR §IV]]`) is platform-agnostic; any per-platform adapter must declare its *threat model* as part of its capabilities surface. A Vegfarendr declaring `surveillance_risk: state-mandated-access` is honest; the operator's consent grant *names* this risk; the agent never delivers to a surveillance-risk channel without an explicit per-message consent that surfaces the risk in plain English. Ember will not ship per-platform adapters that hide this characteristic.

**Vow:** **Surface Without Surveillance** — formalized to include the requirement that every channel-adapter declares its threat model and that delivery consent surfaces it to the operator.

## Refusal 8 — The 11,652-line server.py

**Pattern:** the entire backend is one Python file.

**Evidence:**
- `server.py` is **11,652 lines** by `wc -l`. It contains route handlers, websocket handlers, the dispatch_tool function (`server.py:1032`), the chat completion handler (multiple thousands of lines around `server.py:1500-7300`), the affection system-prompt injection (`server.py:2609-2672`), the VTS prompt injection (`server.py:2580-2607`), the A2UI rendering (`server.py:2675-2700+`), the OpenAI-compat API simulation, the MCP server registration, the VRM WebSocket handlers (`server.py:8181+`), and roughly everything else FastAPI-shaped.

**Why refused:** A single file at 11,652 lines is *not holdable*. No human can carry its model in their head. No AI can edit it safely without thrashing. The Vow of the Unbroken Whole (from `[[hermes:00_OVERTURE]]`) demands wholeness *as held understanding*, not just as uncut delivery. Hermes had the same antipattern at `cli.py:14560`. SAP has it at `server.py:11652`. Both are wrong.

**Ember instead:** The slice-2 ratification (`docs/decisions/0013-second-slice-ratification.md`) and the Mythic Engineering rite of file decomposition discipline keep Ember's files small. The recommendation, reinforced here: any Ember file crossing 1500 lines requires a Decision Record. Any file crossing 800 lines triggers a refactoring review.

**Vow:** Reinforces **Vow of the Unbroken Whole** with the file-size dimension already established in Hermes Codex.

## Refusal 9 — OpenAI-compat simulation as a leaky abstraction

**Pattern:** SAP exposes an OpenAI-compatible REST API that forwards to Claude or Gemini via translation layers, allowing OpenAI-shaped client code to call non-OpenAI models without modification.

**Evidence:**
- `py/ClaudeAsOpenAI.py` — the file's existence is the pattern.
- `py/GeminiAsOpenAI.py` — same.
- `py/dify_openai.py` — Dify-as-OpenAI translation.

**Why refused:** OpenAI-compatible simulation layers are famously leaky. Token counting differs across providers (Anthropic tokens are not OpenAI tokens are not Gemini tokens, and pricing differs accordingly). Tool-call shapes differ subtly (function-calling JSON schema, response format, multi-tool concurrency semantics). System-prompt handling differs (Claude has distinct `system` parameter, OpenAI mixes it into `messages`, Gemini has `systemInstruction`). Stream chunk semantics differ. Streaming-with-tool-calls-with-thinking-blocks is a minefield specific to each provider. Pretending all three are OpenAI lets *most* client code work *most* of the time, and silently breaks the rest in ways the operator cannot diagnose. The Auditor's `[[20_interface/21_OPENAI_COMPAT_API]]` and `[[50_verification/55_API_SIMULATION_TRAPS]]` will catalogue.

**Ember instead:** Strengr (the tether) handles *one cord at a time*. Ember does not pretend to be a multi-provider gateway. If the operator wants Claude, Strengr speaks Claude (Anthropic SDK directly). If the operator wants Gemini, Strengr speaks Gemini. Provider-shape differences are *typed at the Strengr layer*, not papered over by a simulation. Hermes had the same Vow (`[[hermes:00_OVERTURE]]` Strengr is one cord), and this codex confirms it.

## Refusal 10 — Skills installed by zip-fetching from arbitrary GitHub URLs

**Pattern:** the skill installer accepts a GitHub URL, parses owner/repo/branch/subpath, downloads the branch archive, and installs the contents.

**Evidence:**
- `py/skills.py:60-93` — `parse_github_url()` plus `download_zip()`. Accepts any GitHub URL, defaults to `main` branch if not specified, fetches `https://github.com/{owner}/{repo}/archive/refs/heads/{branch}.zip` and installs into `SKILLS_DIR`.
- `py/skills.py:79-88` — calls `https://api.github.com/repos/{owner}/{repo}` to resolve default branch if URL doesn't specify; falls back silently to `main` on failure.

**Why refused:** Arbitrary-URL skill installation is a *supply-chain attack delivery vector*. Any skill installation is essentially "run someone else's code on my machine"; doing it from an arbitrary GitHub URL with a silent main-branch fallback means the operator (or the LLM constructing the URL) can install anything. There is no signature verification, no allowlist, no sandbox, no review surface. The skill becomes part of the agent's tool dispatch.

**Ember instead:** Smiðja (the forge — Ember's tool/skill execution) treats every skill as untrusted code by default. Installation requires explicit operator review of the skill's manifest (declared capabilities, declared file-system access, declared network access). Skills run in a sandbox. The skill registry is operator-curated, not LLM-typeable. The Vow of Honest Memory extends: the agent does not extend its own *toolset* by following URLs the model emitted.

**Vow:** Reinforces **Modular Authorship** with sandbox requirements; folds into the Defended System Prompt Vow's *typed entry surface* concept.

## Refusal 11 — Computer-control via pyautogui with FAILSAFE as the only safety

**Pattern:** the computer-control toolchain uses pyautogui to drive mouse and keyboard, gated only by `pyautogui.FAILSAFE = True` (mouse-corner kill) and a try/except around the import.

**Evidence:**
- `py/computer_use_tool.py:9-29` — `pyautogui.FAILSAFE = True; pyautogui.PAUSE = 0.05`; the `@require_gui` decorator returns a Chinese error string if pyautogui is absent.
- `py/cdp_tool.py` — 559 lines of Chrome DevTools Protocol-driven browser automation.
- `py/cli_tool.py` — 2,668 lines of terminal-driving tools.

**Why refused:** Computer-control by an LLM agent is a *large blast radius* affordance. The model can move the mouse, click anywhere, type anywhere, run terminal commands. The only safety is the operator's mouse-corner failsafe (which works only if the operator is at the keyboard and notices). For an "AI companion" application — defaulted on a desktop where the operator is *not* expected to monitor every action — this is a serious attack surface. Even when the model is well-aligned, the affordance lets a prompt-injection-via-screen-content or via-incoming-IM-message attack take over the operator's computer.

**Ember instead:** Ember does not implement computer-control in slice-3 or earlier. If a future slice considers it, the affordance is gated by typed per-action operator approval (each click, each keypress) with a Vörðr-shaped action-gate (the name-slot from `[[hermes:02_NAMING_PARALLELS §X]]`). The current Vow of Smallness *strongly* discourages this surface at any near-term horizon; the Vow of Surface Without Surveillance and the new **Affective Restraint** Vow together suggest "the agent does not take actions on the operator's behalf without per-action consent" as a Vow worth formalizing.

## Refusal 12 — The behavior engine as singleton with global mutable state

**Pattern:** `BehaviorEngine` is a Python singleton with mutable instance state (handlers, timers, counters) shared across the entire process.

**Evidence:**
- `py/behavior_engine.py:53-99` — `class BehaviorEngine: _instance = None; def __new__(cls): ...` is the canonical Python singleton implementation.
- `py/behavior_engine.py:225` — `global_behavior_engine = BehaviorEngine()` at module load.
- `py/behavior_engine.py:69-73` — `self.platform_activity`, `self.platform_targets`, `self.handlers`, `self.timers`, `self.counters` are mutable dicts on the singleton.

**Why refused:** Global mutable state is the *substrate* for the cross-system tangles that make `server.py:11652` hard to follow. The behavior engine becomes a shared bus between IM bot lifecycles, the affection system, the scheduler, the sub-agent system, and the VRM render windows. Tests cannot isolate the engine; multi-veizla support is structurally impossible (there is *one* engine, so there can be *one* veizla); the singleton's `_initialized` flag (`behavior_engine.py:62-64`) means re-initialization is silently dropped, which masks configuration drift bugs.

**Ember instead:** Ember's Veizla architecture has *one Veizla object per session*, instantiated and disposed cleanly. Behaviors live inside their Veizla, not in a global. Sub-veizlas (Federated Veizla attendant seats) have their own behavior ledgers. The shape of `BehaviorEngine` is adoptable (`[[01_SAP_ESSENCE §VII]]` Adapt); the singleton-ness is not.

## Refusal 13 — Long-term memory + livestream + IM bots all in one process

**Pattern:** the same process that runs the affection-state JSON, the multi-platform IM delivery, and the livestream comment ingestion *also* runs long-term memory providers (mem0, supermemory, m0 — see `server.py:3412, 3629, 3695` etc.).

**Evidence:**
- `server.py:3412` — `if settings["memorySettings"]["is_memory"] and settings["memorySettings"]["selectedMemory"] ...` — memory provider injection into the chat completion path.
- `server.py:3695` — `m0` (mem0) write hook on assistant responses.

**Why refused:** This is a privacy minefield. Every operator interaction across eight IM platforms, three livestreams, the desktop chat, and the AI browser is potentially fed into a long-term memory store. The store is a single JSON-or-database file. The Affection JSON sits next to it. The skill installer can install skills from arbitrary URLs. The computer-control tool runs in the same process. **The combination is the surface a single malicious skill needs to exfiltrate the operator's entire interaction history across every platform.**

**Ember instead:** Ember's storage is *pluggable* (Brunnr) and *operator-scoped*. Long-term memory (if and when Ember has it — the Hugr/Mynd name-slot from Hermes Codex is still reserved) is bounded by the Veizla's declared scope. Cross-channel storage of conversations requires explicit cross-channel consent. The operator can purge per-channel, per-time-range, per-topic.

**Vow:** Reinforces **Affective Restraint** and **Surface Without Surveillance**. The Auditor's `[[50_verification/56_PRIVACY_BOUNDARIES]]` catalogues the threat model in detail.

## Three deferrals

These are SAP patterns Ember should *not* refuse outright but also should *not* adopt soon. They sit in the deferral list until later codex work (or operational experience) clarifies them.

**Deferral A — The extension system.** `py/extensions.py:23-50` exposes a typed `Extension` model with `id`, `name`, `description`, `version`, `author`, `systemPrompt`, `repository`, `backupRepository`, `category`, `transparent`, `width`, `height`, `enableVrmWindowSize`. This is a *better* abstraction than the skill installer (Refusal 10). The pattern is closer to what Ember might eventually want for plugin-shaped extensibility. Defer until `[[60_synthesis/61_NEW_VOWS]]` and the slice plan have addressed sandboxing.

**Deferral B — MCP server exposure.** SAP exposes itself as an MCP server (`README.md:273-282` claims a `http://127.0.0.1:3456/mcp` endpoint; `py/mcp_clients.py:1-189` handles the client side). MCP-as-protocol is genuinely useful and Ember's `[[hermes:00_OVERTURE]]`-era discussion already names it. Defer until Strengr's MCP role is decided in `[[60_synthesis/67_SLICE_PLAN_REVISIONS]]`.

**Deferral C — A2UI.** `server.py:2675-2700+` injects an "A2UI" rendering surface as a system-prompt-tagged JSON format for forms/cards/selects. This is *also* prompt-injection (Refusal 6 pattern repeated for UI) but the *intent* — typed UI rendering from the model's structured output — is salvageable. Defer for a typed-output redesign in `[[60_synthesis/64_AFFECTION_ENGINE_REIMAGINED]]` adjacent work.
